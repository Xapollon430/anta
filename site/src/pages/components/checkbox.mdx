# Checkbox

Checkboxes let the user select one or more items from a set, or turn a single
option on or off.

If you have several options in a list, checkboxes save space compared with a row
of on/off switches. For a single option that takes effect immediately, a switch
reads more clearly than a lone checkbox.

The `Checkbox` wrapper renders an `<a-checkbox>` web component — a small
interactive element that owns its visual state internally, so it works both
**controlled** and **uncontrolled**. The wrapper API is `checked` /
`defaultChecked` (a boolean for the binary axis, `'indeterminate'` for the
third state) and `onStateChange` for change requests. The label is its
`children` (or the `label` prop); `tone` and `size` are plain attributes.

## Basic checkboxes

A checkbox is unchecked, checked, or indeterminate — the last is the "mixed"
state for a parent whose children are partly selected. Indeterminate renders a
minus and takes visual precedence over checked; clicking it resolves to
checked. `disabled` fades the box and drops it from the tab order; it works in
every state.

## Label & hint

Pass the label as `children` (or the `label` prop for the common single-string
case) and the whole control — box plus text — becomes one clickable, focusable
target. Add a `hint` for a secondary line under the label, like Input's hint
text. The label is what `role="checkbox"` takes as its accessible name (no need
to pass `aria-label` yourself — see [Accessibility](#accessibility)).

A label-less, box-only checkbox is supported but is an explicit opt-in: when
the label is genuinely external (e.g. a table-header "select all" with column
context), pass an `aria-label` so the control still has an accessible name.

```tsx
<Checkbox aria-label="Select all rows" />
```

and `large` (20px). The label's type scale tracks the box (13/16, 15/20, 17/24),

## Controlled

By default a checkbox is **uncontrolled** — pass `defaultChecked` (or nothing)
and it flips itself on click; read the value from `onStateChange` or at
form-submit time. To drive it from your own store, make it **controlled** with
`checked` and `onStateChange`: it renders exactly the `checked` you pass, and
`onStateChange` hands you the new state to store.

```tsx
const [agreed, setAgreed] = useState(false)

<Checkbox
  checked={agreed}
  onStateChange={(_e, { next }) => setAgreed(next as boolean)}
>
  I agree
</Checkbox>
```

`onStateChange` is **event-first, detail-second** — `(event, { next, prev })` —
because the event arrives *before* the element applies. A controlled consumer
accepts the change by updating `checked`; a synchronous `event.preventDefault()`
vetoes it in uncontrolled mode (e.g. "confirm before unchecking" without
lifting state).

> A controlled checkbox with no `onStateChange` is effectively read-only — it
> reflects `checked` and ignores clicks (the element never self-applies in
> controlled mode). If a box "won't toggle", it's almost always `checked` with
> no `onStateChange` updating it.

## Indeterminate

A checkbox input only has two states in a form — checked or unchecked — but
*visually* it has three. `checked="indeterminate"` shows the minus glyph and
takes visual precedence; the underlying checked-ness then only affects what a
form submits (an indeterminate box submits nothing, like a native one).

The canonical use is a parent whose state is **derived** from its children:
`true` when both are on, `'indeterminate'` when they differ. Click the
children and watch the parent resolve between the three states.

<Preview>
  <SelectAllDemo client:visible />
</Preview>

```tsx
function ParentChild() {
  const [on, setOn] = useState([true, false])
  return (
    <>
      <Checkbox
        checked={on[0] && on[1] ? true : on[0] || on[1] ? 'indeterminate' : false}
        onStateChange={(_e, { next }) => setOn([next === true, next === true])}
      >
        Parent
      </Checkbox>
      <div style={{ marginInlineStart: 24 }}>
        <Checkbox
          checked={on[0]}
          onStateChange={(_e, { next }) => setOn([next === true, on[1]])}
        >
          Child 1
        </Checkbox>
        <Checkbox
          checked={on[1]}
          onStateChange={(_e, { next }) => setOn([on[0], next === true])}
        >
          Child 2
        </Checkbox>
      </div>
    </>
  )
}
```

## When to use

- **Checkboxes vs. switches** — a checkbox marks an item as selected, usually
  staged until a later "save" or "submit". A switch toggles a setting that takes
  effect immediately. For a single binary option that applies right away, prefer
  a switch; for selecting items in a list, use checkboxes.
- **Multiple independent choices** — when options aren't mutually exclusive (the
  user can pick any combination), use checkboxes. For a single choice out of
  several mutually-exclusive options, use radio buttons instead.

## Accessibility

The `Checkbox` wrapper applies `role="checkbox"` and `aria-checked` (`"true"` /
`"false"` / `"mixed"`) from the state it's given — live for a controlled
checkbox, the initial seed for an uncontrolled one (prefer controlled when a
screen reader must track every toggle). The wrapper also derives the
accessible name from `label` (or a string `children`) and sets it as
`aria-label` on the element automatically — consumers don't pass `aria-label`
by hand for the labelled cases.

The web component itself sets no ARIA; on a raw `<a-checkbox>` you wire it
yourself. It's keyboard-operable: <kbd>Space</kbd> toggles it (funnelled
through the click path).

For a genuinely label-less box (e.g. a table-header "select all" whose meaning
comes from the column), pass an explicit `aria-label`:

```tsx
<Checkbox aria-label="Select all rows" />
```

Follows the [WAI-ARIA checkbox pattern](https://www.w3.org/WAI/ARIA/apg/patterns/checkbox/).

Override any of these on a single instance via `style`, or on a selector wrapping
the checkbox. The box background is the themed `--bg-1` role; the fills, borders,
disabled overlays, and focus ring are per-tone values with a matching dark-mode
block. Dimensional values (box size, radius, label type scale) are baked-in
literals — not tokens — so override them with a plain CSS rule rather than a
variable.

| Token | Description |
|---|---|
| `--checkbox-gap` | Space between the box and the label. |
| `--checkbox-bg` | Unchecked box background. Defaults to the `--bg-1` role. |
| `--checkbox-border`<br />`--checkbox-border-hover`<br />`--checkbox-border-active` | Unchecked box outline across rest / hover / active. |
| `--checkbox-fill`<br />`--checkbox-fill-hover`<br />`--checkbox-fill-active` | Filled (checked / indeterminate) background across rest / hover / active. Re-pointed per `tone`. |
| `--checkbox-icon` | Check / minus glyph color (default white). |
| `--checkbox-icon-disabled` | Glyph color when disabled. |
| `--checkbox-bg-disabled`<br />`--checkbox-border-disabled` | Box fill / outline when disabled. |
| `--checkbox-focus` | Keyboard focus ring color. |
| `--checkbox-label-color`<br />`--checkbox-label-color-disabled` | Label text color (rest / disabled). |
